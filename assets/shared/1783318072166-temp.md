# 开发工作流（必须严格遵守）
## Phase 1 · 动手前：意图对齐
+ 收到需求后先判断是否具备执行条件；
+ 条件不足时必须：① 1-2 句话复述需求 ② 指出歧义点 ③ 给出 3-5 步极简实施计划；
+ **等待用户确认后**才能进入编码阶段。

## Phase 2 · 编码中：小步推进
+ 禁止一次性输出全部代码，按计划逐步完成，每步附说明；
+ 涉及流程/组件交互关系时，先用 Mermaid 画图，再给实现代码；
+ 遇错必溯源：报错时禁止猜测，必须根据日志/调用栈推导原因后再给修改方案。

## Phase 3 · 完成后：三点自检（缺一不可）
1. ✅ 是否完全覆盖需求？有无遗漏边界情况？
2. ✅ 代码是否可直接运行？依赖/配置/环境变量是否完整？
3. ✅ 若修改了已有逻辑，是否评估了对原有功能的影响？

自检通过后按以下格式收尾：

> **✔**** 完成摘要**：做了什么 | 如何验证
>

## Phase 4 · 代码审查
+ 提交审查时主动说明「解决了什么问题」和「潜在风险点」；
+ 接收意见时禁止只改表面，必须解释修改逻辑并确认是否影响关联代码。

## Phase 5 · 上下文管理
+ 对话过长或涉及多个独立模块时，主动建议开启新对话；
+ 涉及 Git 破坏性操作（reset / force push）时，必须给出警告并等待确认。

# 当前任务：有限空间作业审批表全栈实现
## 一、任务目标（一句话）
在 **校领导端** 基于既有 BPM 能力，为「有限空间安全作业审批表」实现完整的增删改查 + 打印 + BPM 发起/审批/**退回发起人可重新编辑** 全流程，前后端表单样式与下方附件 Word 文档、打印件保持一致。

## 二、已确定的数据库结构（既定事实，不在本次任务中重新设计）
> 该 SQL 已根据业务需求评审确定，直接作为开发依据。若开发中发现字段无法满足前端展示/交互需求，**先提出变更点走确认流程，不擅自改表**。
>

```sql
-- 主表：leader_confined_space_work_approval（有限空间作业审批表）
-- 明细表：leader_confined_space_work_approval_gas（有毒有害气体/粉尘检测项，动态增行）
```

关键字段口径（供编码时对照，避免遗漏）：

+ `work_unit`（作业单位）、`affiliated_unit`（所属单位）：**必填**；
+ `main_hazard_factors` / `work_content` / `work_personnel`：文本域（text）；
+ `filler` / `guardian` / `test_personnel`：varchar；
+ `test_time` / `work_start_time`：日期+时间（datetime）；
+ `oxygen_content` / `flammable_gas_concentration`：varchar；
+ 有毒有害气体(粉尘)浓度：走明细表 `leader_confined_space_work_approval_gas`，前端支持动态增/删行；
+ `safety_measure_worker_confirm` / `safety_measure_guardian_confirm`：十条安全措施签名确认，**非必填**，JSON 数组存储，允许留空/手写展示，不做强校验；
+ `work_leader_opinion`：发起人手写意见（非 BPM 节点，普通表单字段）；
+ `work_leader_sign_user_id` / `work_leader_sign_user_name` / `work_leader_sign_time`：提交时**后端自动获取当前登录人**写入，前端不提供手动输入；签名图片由后端已有的"根据用户ID取签名图"能力渲染，**本次无需再实现签名图存储逻辑**；
+ `process_instance_id` / `status` / `dept_id` / `dept_name` / `applicant` / `apply_date`：BPM 相关，表单底部「申请单位/安全保卫处/分管安全领导/校长审批意见」**完全走 BPM 节点意见（act_hi_comment 等）**，业务表不冗余存储审批意见。

## 五、必须实现的「退回发起人可重新编辑」机制
本模块需复用项目已铺设好的 BPM 容器通用底座，按"两步走"模板接入：

**第一步 · 前端业务表单（create.vue）**

+ 通过 `inject('bpmTodoTask')` / `inject('bpmProcessInstance')` 判断是否处于发起人退回重做态（`isInitiatorRedo`），兼容 `taskKey === 'initiatorTask' | 'StartUserNode'` 以及任务名包含"发起人/重做/重新提交"的情况，并校验当前用户是否为流程发起人本人；
+ 监听 `detailId`（优先取 `bpmProcessInstance.businessKey`，否则取 props.id），重做态下自动调用 `getDetail` 回显历史数据（含明细表 `gasList` 的防御性格式化）；
+ `defineExpose({ resubmitForm })`，内部先校验表单，再调用 `update` 接口提交修改。

**第二步 · 后端明细表重建**

+ 明细保存方法中，重新插入前**强制将明细 **`id`** 置为 **`null`，避免逻辑删除后主键冲突：

```java
private void createGasDetailList(Long approvalId, List<ConfinedSpaceWorkApprovalGasDO> list) {
    if (CollUtil.isEmpty(list)) {
        return;
    }
    list.forEach(o -> {
        o.setApprovalId(approvalId);
        o.setId(null); // 强制置空，避免退回重提时明细主键冲突
    });
    gasMapper.insertBatch(list);
}
```

**BPM 流程配置前提**：需确认该流程在 BPM 流程定义管理中已正确填写 `formCustomCreatePath`（编辑页）与 `formCustomViewPath`（只读详情页），且详情容器 `views/bpm/processInstance/detail/index.vue` 中的 `isInitiatorRedo` 判定已按通用底座逻辑放宽（若尚未放宽，需先确认这是否属于本次任务范围，还是已由其他任务完成）。
